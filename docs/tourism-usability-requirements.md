# J-Connect 観光記事: 実用性改善要件

基準コミット: `0fba4813d386f81233e4e85fa44862aa01d36530`（PR #299 merge）

対象は `category: "観光"` の全20記事。写真追加はこのPRの対象外とし、以下の実用性改善を優先する。ユーザーの指摘②は内容が空欄だったため、推測で要件を追加しない。

## 1. 「公式サイトを確認」で読者を放置しない

主要な予約・運行・入場条件について、リンクだけで終わらせず、次を短く示す。

- 公式ページを開いた後、どのメニュー名・ボタン名を見るか
- ドイツ語または英語の画面上で探す語句
- 日付、時刻、販売状況、運休など、最終的に何を確認すればよいか
- ブラウザ翻訳を使う場合でも、日付・時刻・固有名詞・購入条件は原文表示へ戻して照合すること

例:

> **市庁舎内部見学の確認手順**：公式ページを開き、`Guided tours` または `Führungen` を選ぶ。カレンダーで訪問日を開き、開始時刻、言語、残席、集合場所を確認する。ブラウザ翻訳を使った後も、予約画面の日時と人数は原文で照合する。

すべての外部リンクに同じ説明を付ける必要はない。旅程を固定する主要リンクに絞り、記事ごとに2〜4か所を具体化する。

## 2. 読者像を「ドイツ在住日本人」に戻す

- DB Navigator、Deutschlandticket、ドイツ国内のS-Bahn/U-Bahn等の基礎は共通ガイドへ集約する。
- 国内観光記事で、刻印・検札・ICE/ICの説明を毎回長く繰り返さない。
- 現地固有の例外だけを1〜3文で説明し、共通ガイドの該当アンカーへ内部リンクする。
- ドイツ国外の記事では、ドイツの制度と違う点だけを比較して説明する。
- 「初めてヨーロッパへ来た旅行者」向けのパスポート・カード・基本マナーの一般論で記事を膨らませない。

このPRで次の共通記事を先に新設する。

1. `/germany/ja/living/germany-train-travel-guide/`
   - タイトル案: `ドイツの電車の乗り方｜DB Navigator・Deutschlandticket・長距離列車の基本`
   - 直接リンクできる見出し: DB Navigator、列車種別、Deutschlandticket、市内交通券と刻印、工事・代行バス、検札、子連れ・バリアフリー
2. `/germany/ja/learn-german/travel-german-phrases/`
   - タイトル案: `観光・週末旅行で使えるドイツ語｜駅・ホテル・観光案内所のフレーズ`
   - 直接リンクできる見出し: 駅、遅延・運休、ホテル、観光案内所、予約・入場、聞き返し

## 3. 内部リンクを文脈のある位置に置く

- 冒頭で他都市を勧めるなら、その都市記事へその場でリンクする。
- 電車・交通の共通知識に触れる最初の場所から、鉄道共通ガイドの該当アンカーへリンクする。
- まとめ直前に、観光で使えるドイツ語記事へ短い準備導線を置く。
- 記事末尾の関連記事だけに内部リンクを集中させない。
- リンク文言を `こちら` にせず、リンク先で得られる内容を明示する。
- `related_articles` は本文内リンクを補完するものとして整理し、同じリンクを不自然に重複させない。

## 4. 位置関係・基本ルートを前半で図示する

全20記事に、本文前半で見られるオリジナル概略図を1つ追加する。

- 見出し: 原則 `位置関係と基本ルート`
- 中央駅・到着地点、主な観光地点、川・旧市街などの位置関係を示す
- 基本ルートは実線、任意の追加・雨天分岐は破線などで区別する
- 北/南など方角を表示する場合は実際の地理と一致させる
- 縮尺図ではないことを図とキャプションの両方で明記する
- ノード名から該当する本文見出しへ移動できる設計を優先する
- デスクトップ用820×520前後、モバイル用480×720前後を生成し、モバイルで文字を縮小し過ぎない
- 実写地図、Google Maps等のスクリーンショット、第三者のイラスト地図、地図タイルの転載は禁止
- 外部スクリプト、外部フォント、位置情報、オンライン地図への依存は禁止
- 見た目だけでなく、alt、SVGの`title`/`desc`、十分なコントラスト、キーボード操作を検証する

## 5. 読者を責める見出しをやめる

次のような見出しを全20記事からなくす。

- よくある失敗
- やりがちな失敗
- ありがちな失敗
- 典型的な失敗
- 起こりやすい失敗
- 港で起きやすいミス

原則 `注意点のまとめ`、必要なら `出発前に確認したいポイント`、`旅程を崩さないための注意点` などへ変更する。本文も断罪調ではなく、「〜すると時間を失います」から「〜を先に済ませると移動を短くできます」のように行動提案へ直す。

## 6. 記事の締めを整える

全20記事の本文末尾に `## まとめ` を置く。

- 2〜4段落で、その目的地を選ぶ価値、最も重要な判断、天候や同行者による変更を自然に振り返る
- チェックリストを言い換えただけの箇条書きにしない
- 新情報を突然追加しない
- `まとめ` の後に、生成側の「公式情報・参考ソース」「関連記事」が続く順番に統一する
- Markdown本文中の重複した `## 関連記事` は原則削除し、frontmatterの `related_articles` と生成UIへ集約する

## 7. 公式ソース・関連記事のリスト表示を修正する

現在は全体のCSSリセット後、記事外側の `.related-section ul` に適切な左余白と項目間隔が戻されていない。

- `.official-source-section` と `.related-section` の `ul`、`li` を明示的に整える
- 箇条書き記号がカード外にはみ出さない
- desktop/mobile、light/darkで余白とコントラストを確認する
- HTML文字列のインデントだけで直したことにせず、実際の描画をbrowser testで検証する

## 全記事の合格条件

- 概略ルート図が本文の前半25%以内に1つある
- `失敗` または `ミス` を含む注意見出しが0件
- `## まとめ` が1件あり、本文最後のh2である
- 文脈内の内部リンクが最低3件。ただし関連性のないリンク数稼ぎは禁止
- 鉄道共通ガイドと旅行ドイツ語記事へのリンクが適切な文脈にある
- 主要公式リンク2〜4件に、ページ上で何を見るかの短い手順がある
- 公式ソースと関連記事の箇条書きがdesktop/mobileで正しく字下げされる
- 目次はdesktopでは右サイドバー、mobileでは既存の折りたたみ式。中央列へ別の目次を追加しない
